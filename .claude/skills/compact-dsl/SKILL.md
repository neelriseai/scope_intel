## Concept — Compact DSL for Architectural Knowledge

### The Problem It Solves

Storing design knowledge as prose markdown is expensive at read time:

```markdown
## Memory Layer
The memory layer is responsible for storing and retrieving information across
sessions. It uses Redis for short-term session memory with a TTL of 24 hours.
For long-term persistence it uses PostgreSQL with pgvector for semantic search.
It exposes three interfaces: store(), retrieve(), search(). It must never store
PII directly and all session keys must carry a TTL.
```
**~150 tokens** — Claude reads this every session for every feature.

Same knowledge in compact DSL:
```
[memory-layer|module|high]
uses: Redis(role=session,ttl=24h), PostgreSQL(ext=pgvector,role=long-term)
api:  store(key,val,ttl?), retrieve(key), search(query,top_k=5)
rule: no-PII, ttl-required-on-session-keys, encrypt-at-rest
deps: → engine, ← rag-layer
phase: 2
```
**~45 tokens** — same semantics, 70% compression.

## DSL Grammar (proposed)

### Component block
```
[id | type | importance | status?]
purpose: one-line description
uses:    dep(attr=val, ...), dep2(...)
api:     method(params) → return_type, ...
rule:    constraint1, constraint2
pattern: architectural-pattern-name
deps:    → outgoing-component, ← incoming-component
phase:   N
```

### Fact line
```
!FACT      [subject | predicate | object | conf=0.0-1.0]
!CONSTRAINT[rule text | scope=component]
!DECISION  [what was decided | reason=why | date=YYYY-MM]
!FLOW      [trigger → step1 → step2 → output]
```

### Module map entry
```
component-id: src/path/*.py → [symbol1, symbol2] | owner=team
```

### Type/Schema entry
```
@Type  TypeName { field: type, field2: type? }
@Enum  StatusEnum { pending | running | done | failed }
```

## How It Fits the Doc Ingest Pipeline

### Option A — LLM outputs DSL directly
```
Qwen prompt: "Extract this section in this DSL format: [id|type|importance]..."
Qwen output: DSL text
Code:         DSL parser → structured dict → write .scope files
```
**Pro:** Fewest tokens generated by LLM (fast, cheap)
**Con:** LLM can hallucinate DSL syntax — needs validation + repair loop

### Option B — LLM outputs JSON → code converts to DSL (recommended)
```
Qwen output: {"section_title": "Memory Layer", "category": "module", ...}
Code:         JSON → DSL renderer → write .scope files
```
**Pro:** JSON output is reliable (Ollama format=json enforces it)
**Con:** Slight overhead converting JSON → DSL, but code-side, not LLM-side

### Option C — DSL as storage, prose preserved separately
```
.ai-context/generated/005-memory-layer.scope   ← DSL (Claude primary read)
.ai-context/generated/005-memory-layer.md      ← prose (human reference)
```
**Pro:** Best of both worlds
**Con:** Two files per component, more to maintain

## Where DSL Helps Most in This Repo's Problem

| Stage | Without DSL | With DSL | Saving |
|---|---|---|---|
| LLM extraction output | JSON ~800 tokens/chunk | DSL ~250 tokens/chunk | ~69% |
| Claude reads per session | MD ~2000 tokens/feature | DSL ~600 tokens/feature | ~70% |
| mempalace.jsonl entries | Full prose notes | `!FACT[...]` lines | ~60% |
| module-map.md | Prose paragraphs | `id: path → [syms]` | ~75% |

---

## Where DSL Hurts

- **Design rationale / decisions** — nuance is lost in compression.
  Decisions need prose: *why* was Redis chosen? DSL can't capture that well.
- **New team member readability** — DSL requires learning the grammar.
- **LLM schema drift** — if Qwen doesn't follow grammar exactly, parser breaks.
  Mitigation: always validate + fall back to JSON.
- **DSL evolution cost** — if grammar changes, all stored `.scope` files need migration.

---

## Recommendation for Doc Ingest

Use DSL **selectively**, not everywhere:

```
Use DSL for:                    Use prose for:
─────────────────────────────   ──────────────────────────────
module-map.md                   001-project-overview.md
!FACT / !CONSTRAINT lines       decisions.md (rationale)
component dependency graph      current-phase.md (narrative)
API signature listing           roadmap.md (context-rich)
symbol schema types
```

Hybrid approach:
```
# 005-memory-layer.md (generated file)

<!-- DSL header — Claude reads this block first (low token) -->
```scope-dsl
[memory-layer|module|high|phase=2]
uses: Redis(role=session,ttl=24h), PostgreSQL(ext=pgvector)
api:  store(), retrieve(), search()
rule: no-PII, ttl-required
deps: → engine, ← rag-layer
```

## Memory Layer — Detail

The memory layer handles... (prose for humans and deeper context)
```

Claude reads the DSL block first (~45 tokens), skips prose unless it needs detail.

---

## Relation to Page Indexing

Page indexing + DSL work well together:
- Each DSL block can carry `src_page: 23-25` — traceability back to original doc
- If Claude needs the full context, it can request the original page range
- Compact representation + full source available on demand

---

## Decision Gate — When to Use This Technique

Use compact DSL if:
- [ ] Scope index files are growing large (>500 tokens per component file)
- [ ] Sessions consistently hit token budget on context loading
- [ ] Team is comfortable maintaining a DSL grammar
- [ ] LLM (Qwen/Ollama) reliably follows JSON schema (validate first)

Skip compact DSL if:
- [ ] Team members need to read .ai-context files directly without tooling
- [ ] Document types are highly narrative (rationale-heavy)
- [ ] Single-developer project (overhead not worth it)
